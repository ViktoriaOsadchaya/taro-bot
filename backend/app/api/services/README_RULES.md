# Правила написания Сервисов (Service Layer)

Сервисный слой является ядром бизнес-логики приложения. Все сервисы должны следовать единому стандарту для обеспечения чистоты, тестируемости и поддержки кода.

## Общие принципы
**НЕ ПИСАТЬ ЛОКАЛЬНЫЕ ИМПОРТЫ**
### 1. SOLID и чистая архитектура
*   **Single Responsibility (S)**: Один сервис отвечает за одну доменную область (например, `MilestoneService`). Не смешивайте логику разных сущностей в одном классе.
*   **Dependency Inversion (D)**: Сервис никогда не инстанцирует зависимости сам. Все зависимости (репозитории, сессия, другие сервисы) передаются через конструктор (**Dependency Injection**).
*   **Interface Segregation (I)**: Внедряйте только те репозитории и сервисы, которые действительно необходимы для работы.

### 2. Запрет SQL-запросов в сервисах
*   **Никакого SQL**: В сервисном слое строго запрещено написание SQL-запросов (`select`, `insert`, `update`, `delete`, `text()` и т.д.).
*   **Использование репозиториев**: Для любого взаимодействия с базой данных должны использоваться исключительно методы репозиториев. Если нужного метода в репозитории нет, его необходимо там создать.
*   **Делегирование**: Если логика требует данных из другой доменной области, используйте соответствующий сервис этой области.

### 3. Dependency Injection (Dishka)
Сервисы реализуются в виде классов. Все зависимости (сессия, репозитории, другие сервисы) внедряются через конструктор (`__init__`). Для интеграции с FastAPI используется Dishka.

#### Пример конструктора:
```python
class MilestoneService:
    def __init__(
        self,
        session: AsyncSession,
        repo: MilestoneRepository,
        project_repo: ProjectRepository
    ):
        self.session = session
        self.repo = repo
        self.project_repo = project_repo
```

#### Настройка провайдера:
В конце файла сервиса (или в специальном модуле) создайте провайдер:
```python
class MilestoneServiceProvider(Provider):
    @provide(scope=Scope.REQUEST)
    def milestone_service(
        self,
        session: AsyncSession,
        repo: MilestoneRepository,
        project_repo: ProjectRepository
    ) -> MilestoneService:
        return MilestoneService(session, repo, project_repo)
```
Зарегистрируйте провайдер в `app/core/container.py`.

### 4. Обработка ошибок
*   Используйте кастомные исключения из `app.core.exceptions`.
*   Сервис должен перехватывать технические ошибки БД (если нужно) и выбрасывать понятные бизнес-исключения.

## Правила оформления кода

### 1. Форматирование (Clean Code)
*   **Никаких лишних пробелов**: Строго запрещены пробелы в пустых строках (ошибка W293). Пустая строка должна быть абсолютно пустой.
*   **Одна пустая строка** между методами внутри класса.
*   **Две пустые строки** между импортами и определением класса.
*   **Понятность кода**: Код должен быть максимально понятным и прозрачным для человека. Если участок кода неочевиден, сложен или может быть трактован неоднозначно, его необходимо прокомментировать.

### 2. Импорты
*   **Только top-level импорты**: Все `from ... import ...` должны находиться в начале файла.
*   **ЗАПРЕЩЕНЫ локальные импорты** внутри методов или конструкторов.
*   Группируйте импорты: сначала стандартные библиотеки, затем сторонние (fastapi, sqlalchemy), затем внутренние модули проекта.

### 3. Логика методов
*   Методы должны быть по возможности короткими и решать одну задачу.
*   **СТРОГО ЗАПРЕЩЕНО** прямое написание SQL-запросов или использование SQLAlchemy-конструкторов запросов (`select`, `join` и т.д.) внутри сервиса.
*   Используйте методы репозиториев для получения и сохранения данных.
*   Используйте DTO для входных и выходных данных.

### 4. Комментарии к классам
*   В начале каждого сервисного класса обязателен поясняющий комментарий (docstring), который отвечает на 3 вопроса:
    1. Что это за класс.
    2. Для чего он нужен (какую бизнес-задачу решает).
    3. Где используется (например, в каких роутах/модулях или другими сервисами).

### 5. Комментарии к методам
*   Каждый метод сервисного слоя должен быть прокомментирован docstring в тройных кавычках `""" """`.
*   Docstring должен кратко объяснять, что делает метод и какой бизнес-сценарий он обслуживает.

```python
async def process_tender(self, tender_id: int) -> None:
    """Обрабатывает тендер по идентификатору."""
```

## Чек-лист при создании сервиса:
- [ ] Сервис является классом.
- [ ] В начале класса есть поясняющий комментарий: что за класс, для чего нужен, где используется.
- [ ] Каждый метод сервиса содержит docstring в тройных кавычках `""" """`.
- [ ] Все зависимости (включая `session`) передаются в `__init__`.
- [ ] Создан и зарегистрирован `ServiceProvider` для Dishka.
- [ ] Нет локальных импортов.
- [ ] Нет лишних пробелов в пустых строках.
- [ ] Неочевидные участки кода прокомментированы и понятны человеку.
- [ ] Используются бизнес-исключения.
- [ ] Вся работа с БД идет строго через методы репозитория (в сервисе нет `select`, `update` и т.д.).
