# Правила Написания Роутеров (API Layer)

Документ фиксирует обязательные правила разработки роутеров в `app/api/routers`.

## 1. Назначение роутера
- Роутер отвечает только за HTTP-слой: входные параметры, dependency, вызов сервиса, возврат DTO.
- Бизнес-логика в роутере запрещена. Вся логика должна жить в сервисах.
- SQL-запросы в роутере запрещены.

## 2. Структура файла
- В начале файла: `from __future__ import annotations`.
- Импорты только top-level, без локальных импортов внутри функций.
- Обязателен `APIRouter` с явными `prefix`, `tags`, `route_class=DishkaRoute` (если используется Dishka).
- Имена handler-функций должны быть глагольными и однозначными (`get_...`, `list_...`, `create_...`, `update_...`, `delete_...`).

## 3. Dependency и безопасность
- Для каждого endpoint должна быть явная модель доступа:
- через `dependencies=[...]` на декораторе;
- и/или через параметры-зависимости в сигнатуре.
- Нельзя добавлять endpoint без проверки доступа, если он не задумано публичный.
- Если endpoint публичный, это должно быть явно отражено в комментарии и в `README.md` карты безопасности.

## 4. Контракты API
- Для каждого endpoint обязателен `response_model`.
- Входные параметры должны быть строго типизированы через `Annotated + Path/Query/Body`.
- Для пагинации использовать единые параметры:
- `skip: int = Query(ge=0)`
- `limit: int = Query(ge=1, le=100)` (или другой согласованный верхний порог).
- Ответы должны возвращаться только через DTO, а не сырые ORM-объекты.

## 5. Стиль комментариев (docstring)
- У каждого handler должен быть docstring в формате:
- краткое назначение;
- блок «Параметры» со списком аргументов;
- блок «Возвращает» с типом ответа.
- Комментарии пишутся на русском языке.
- Пример формата:

```python
"""
Получение общего расширенного списка сущностей.

Параметры:
- **entity_pk** (int): Идентификатор сущности.
- **skip** (int): Количество пропускаемых записей.
- **limit** (int): Количество вычитываемых записей.

Возвращает:
- **list[EntityReadDTO]**: Расширенный список сущностей.
"""
```

## 6. Обработка ошибок
- Роутер не конструирует бизнес-ошибки вручную, если это зона сервиса.
- Сервис должен выбрасывать доменные исключения (`app.core.exceptions`), а роутер только делегирует вызов.
- В роутере допускаются только базовые валидации HTTP-уровня (`Path`, `Query`, `Body`).

## 7. Изменения и сопровождение
- При добавлении нового роутера:
- подключить его в `app/api/router.py`;
- добавить/обновить запись в `app/api/routers/README.md` (карта безопасности).
- При изменении контрактов endpoint обновлять DTO и тесты маршрутов.

## 8. Чек-лист перед merge
- [ ] Endpoint имеет корректный `response_model`.
- [ ] Endpoint защищен нужной dependency.
- [ ] В роутере нет бизнес-логики и SQL.
- [ ] Добавлен docstring по шаблону «Параметры/Возвращает».
- [ ] Добавлены или обновлены route-тесты.
- [ ] Обновлен `app/api/routers/README.md` при изменении surface API.
