# Правила написания Репозиториев (Repository Layer)

Репозитории отвечают исключительно за взаимодействие с базой данных. Они скрывают детали реализации запросов SQLAlchemy от остального приложения.

## Общие принципы
**НЕ ПИСАТЬ ЛОКАЛЬНЫЕ ИМПОРТЫ**
### 1. Наследование от BaseRepository
Все новые репозитории **ОБЯЗАНЫ** наследоваться от `BaseRepository[Model]`. Это дает доступ к стандартным CRUD методам (`get_by_id`, `create`, `update`, `delete`, `get_paginated`) «из коробки».

```python
class MilestoneRepository(BaseRepository[Milestone]):
    def __init__(self):
        super().__init__(Milestone)
```

### 2. SOLID
*   **Single Responsibility (S)**: Репозиторий должен содержать только логику запросов к БД. Никакой бизнес-логики, отправки уведомлений или сложной валидации.
*   **Open/Closed (O)**: Репозиторий открыт для расширения через добавление специфичных методов (например, `get_milenstone_for_period`), но закрыт для изменения базового поведения CRUD.

### 3. Инъекция через Dishka
В новой архитектуре мы уходим от прямого использования глобальных инстансов. Каждый репозиторий должен иметь соответствующий `Provider`:
```python
from dishka import Provider, Scope, provide

class MilestoneRepoProvider(Provider):
    scope = Scope.REQUEST

    @provide
    async def get_repo(self) -> MilestoneRepository:
        return milestone_repository  # или MilestoneRepository()
```
Глобальный экземпляр `milestone_repository = MilestoneRepository()` сохраняется только для обеспечения работы старого (deprecated) кода.

## Правила оформления кода

### 1. Форматирование
*   **Strict No Whitespace**: В пустых строках не должно быть пробелов. Это критично для поддержания чистоты репозитория.
*   Соблюдайте стандарты PEP8.
*   **Понятность кода**: Код должен быть максимально понятным и прозрачным для человека. Если участок кода неочевиден, сложен или может быть трактован неоднозначно, его необходимо прокомментировать.

### 2. Методы и сигнатуры
*   Каждый метод репозитория первым аргументом всегда принимает `session: AsyncSession`.
*   Каждый метод репозитория должен быть прокомментирован docstring в тройных кавычках `""" """`.
*   Используйте типизацию для всех аргументов и возвращаемых значений.
*   Старайтесь использовать `**kwargs` в `create` и `update` методах для обеспечения гибкости и совместимости с `BaseRepository`.

```python
async def get_by_tender_id(self, session: AsyncSession, tender_id: int) -> Tender | None:
    """Возвращает тендер по идентификатору."""
```

### 3. Сложные запросы
*   Сложная логика (JOIN, агрегация, специфичные условия WHERE) должна быть инкапсулирована внутри методов репозитория.
*   Возвращайте из репозитория либо объекты моделей, либо Scalar-результаты. Не возвращайте сырые кортежи (Row), если это не продиктовано производительностью.

## Правила рефакторинга со старой системы

При переводе старого функционала (на функциях) на новую систему (классы + Dishka):

1. **Создайте класс репозитория**: Перенесите все функции из файла в методы этого класса.
2. **Наследование**: Убедитесь, что класс наследуется от `BaseRepository`.
3. **Первый аргумент**: Переименуйте первый аргумент функций `session` на `self`, но оставьте `session` вторым аргументом, если это метод (в методах `BaseRepository` сессия передается в каждый вызов).
4. **Совместимость**: Оставьте старые функции в конце файла как "прокси" к методам класса для обратной совместимости.
5. **Dishka**: Создайте `RepoProvider` и зарегистрируйте его в `app/core/container.py`.

## Чек-лист при создании репозитория:
- [ ] Наследование от `BaseRepository[T]`.
- [ ] Только логика запросов к БД (никакой бизнес-логики).
- [ ] Создан `Provider` для Dishka.
- [ ] Экспортирован список провайдеров в `container.py`.
- [ ] Нет локальных импортов.
- [ ] Пустые строки не содержат пробелов.
- [ ] Сигнатуры методов типизированы.
- [ ] Каждый метод содержит docstring в тройных кавычках `""" """`.
- [ ] Неочевидные участки кода прокомментированы и понятны человеку.

## Дополнительное правило по комментариям входных параметров
- Для методов с большим количеством входных параметров обязательно добавляйте в теле метода заметные комментарии по ключевым аргументам в формате `***param: что делает***`.
- Цель: чтобы параметр и его роль можно было быстро найти и визуально выделить при чтении кода.
